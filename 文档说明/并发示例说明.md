## 🎯 实现的并发问题

### 1. 生产者-消费者问题（单缓冲区）

**问题描述：**
一个生产者和一个消费者共享一个缓冲区。生产者生产产品放入缓冲区，消费者从缓冲区取出产品消费。

**同步约束：**
- 缓冲区满时，生产者必须等待
- 缓冲区空时，消费者必须等待
- 生产者和消费者不能同时访问缓冲区

**信号量设置：**
```python
mutex = 1   # 互斥信号量，保护缓冲区
empty = 1   # 空缓冲区数，初值1
full = 0    # 满缓冲区数，初值0
```

**关键代码逻辑：**
```python
# 生产者
P(empty)      # 等待空缓冲区
P(mutex)      # 进入临界区
放入产品
V(mutex)      # 离开临界区
V(full)       # 增加满缓冲区数

# 消费者
P(full)       # 等待满缓冲区
P(mutex)      # 进入临界区
取出产品
V(mutex)      # 离开临界区
V(empty)      # 增加空缓冲区数
```

**关键要点：**
- ⚠️ P操作顺序：先资源信号量（empty/full），后互斥信号量（mutex）
- ⚠️ V操作顺序：先互斥信号量（mutex），后资源信号量（empty/full）
- 如果顺序错误，可能导致死锁

### 3. 读者-写者问题（读者优先）

**问题描述：**
多个读者和写者共享一个数据对象。多个读者可以同时读，但写者写时不能有其他读者或写者。

**同步约束：**
- 多个读者可以同时读
- 写者独占访问（写时不能有读者或其他写者）
- 读者优先策略：有读者在读时，新来的读者可以直接进入

**信号量设置：**
```python
wmutex = 1      # 写互斥信号量
rmutex = 1      # 读者计数器互斥信号量
readcount = 0   # 读者计数器
```

**关键代码逻辑：**
```python
# 读者
P(rmutex)
readcount++
if (readcount == 1):
    P(wmutex)  # 第一个读者锁定写操作
V(rmutex)

读数据

P(rmutex)
readcount--
if (readcount == 0):
    V(wmutex)  # 最后一个读者释放写锁
V(rmutex)

# 写者
P(wmutex)      # 独占访问
写数据
V(wmutex)
```

**特点：**
- 读者优先：只要有读者在读，新来的读者可以直接进入
- 缺点：可能导致写者"饥饿"（长期得不到服务）

**测试结果：**
- 3个读者 + 2个写者
- 读取9次，写入4次
- 数据一致性验证通过 ✓

### 4. 哲学家就餐问题（避免死锁）

**问题描述：**
5个哲学家围坐在圆桌旁，每人面前有一根筷子。哲学家需要拿起左右两根筷子才能进餐。

**死锁风险：**
如果每个哲学家同时拿起左边的筷子，然后等待右边的筷子，就会发生死锁。

**解决方案：资源编号法**
- 给每根筷子编号（0-4）
- 哲学家按编号顺序获取筷子：先拿编号小的，再拿编号大的
- 破坏了"循环等待"条件，避免死锁

**关键代码逻辑：**
```python
left = philosopher_id
right = (philosopher_id + 1) % num_philosophers

# 关键：按编号顺序获取资源
first = min(left, right)
second = max(left, right)

P(chopsticks[first])   # 先拿编号小的筷子
P(chopsticks[second])  # 再拿编号大的筷子

进餐

V(chopsticks[first])   # 放下筷子
V(chopsticks[second])
```

**为什么能避免死锁：**
- 哲学家0：拿筷子0和1（先0后1）
- 哲学家1：拿筷子1和2（先1后2）
- 哲学家2：拿筷子2和3（先2后3）
- 哲学家3：拿筷子3和4（先3后4）
- 哲学家4：拿筷子0和4（先0后4）⚠️

注意哲学家4会先拿筷子0，而不是筷子4，这打破了循环等待链。

**测试结果：**
- 5个哲学家，每人进餐3次
- 无死锁发生 ✓
- 执行时间正常（<10秒）✓

## 🔑 关键知识点总结

### 1. 信号量的物理意义

```
S > 0：表示可用资源数
S = 0：表示无可用资源，但无进程等待
S < 0：|S| 表示等待该资源的进程数
```

### 2. P/V操作定义

**P操作（wait/down）：**
```
P(S):
    S = S - 1
    if (S < 0):
        阻塞该进程，并将其插入等待队列
```

**V操作（signal/up）：**
```
V(S):
    S = S + 1
    if (S <= 0):
        唤醒等待队列中的一个进程
```

### 3. 信号量类型

1. **互斥信号量（mutex）**
   - 初值：1
   - 用途：保护临界区，实现互斥访问

2. **资源信号量**
   - 初值：资源数量
   - 用途：管理有限资源的分配

3. **同步信号量**
   - 初值：0
   - 用途：实现进程间的同步

### 4. 死锁的四个必要条件

1. **互斥条件**：资源不能共享
2. **请求和保持条件**：持有资源的同时请求新资源
3. **不可剥夺条件**：资源不能被强行剥夺
4. **循环等待条件**：存在进程资源的循环等待链

**避免死锁的方法：**
- 破坏任意一个必要条件即可
- 本项目使用"资源编号法"破坏循环等待条件

### 5. 常见并发错误

❌ **错误1：P操作顺序错误**
```python
# 错误：先互斥后资源
P(mutex)
P(empty)  # 可能死锁！
```

✓ **正确：先资源后互斥**
```python
P(empty)
P(mutex)
```

❌ **错误2：忘记释放信号量**
```python
P(mutex)
# 操作临界区
# 忘记V(mutex) - 导致其他进程永久阻塞
```

❌ **错误3：信号量初值设置错误**
```python
empty = 0  # 错误！应该是缓冲区大小
full = n   # 错误！应该是0
```

## 📊 性能分析

### 单缓冲区 vs 多缓冲区

**单缓冲区：**
- 生产者和消费者严格交替执行
- 并发度低
- 适合简单场景

**多缓冲区：**
- 生产者和消费者可以并发执行
- 并发度高
- 吞吐量提升明显

**实测数据（缓冲区大小=5）：**
- 2个生产者 + 2个消费者
- 生产10个产品耗时：约1-2秒
- 缓冲区利用率：平均20-40%

### 读者优先 vs 写者优先
