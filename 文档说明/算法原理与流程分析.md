# 操作系统算法原理与流程分析

根据五章总结内容，详细分析每个算法的原理和执行流程。

---

## 第二章 处理器管理 - 调度算法

### 1. 先来先服务（FCFS, First Come First Served）

**算法原理：**
- 按照作业或进程到达的先后顺序进行调度
- 先到达的先得到服务，后到达的排队等待
- 非抢占式调度算法

**算法流程：**
1. 作业/进程到达系统，加入就绪队列尾部
2. CPU空闲时，从就绪队列头部取出第一个作业/进程
3. 该作业/进程占用CPU直到完成或阻塞
4. 完成后，继续从队列头部取下一个
5. 重复步骤2-4

**优点：**
- 实现简单
- 公平，不会产生饥饿

**缺点：**
- 平均等待时间波动大，取决于作业到达顺序
- 对短作业不利
- CPU和I/O设备利用率低

**示例：**
```
作业序列：A(8ms) -> B(4ms) -> C(2ms)
执行顺序：A -> B -> C
周转时间：A=8, B=12, C=14
平均周转时间：(8+12+14)/3 = 11.33ms
```

---

### 2. 最短作业优先（SJF, Shortest Job First）

**算法原理：**
- 优先调度预计运行时间最短的作业
- 以作业的预估执行时间作为优先级
- 可以是抢占式（SRTF）或非抢占式

**算法流程：**
1. 所有就绪作业按预估运行时间排序
2. 选择运行时间最短的作业执行
3. 作业完成后，重新从剩余作业中选择最短的
4. 重复步骤2-3直到所有作业完成

**优点：**
- 平均等待时间最短（理论最优）
- 提高系统吞吐量

**缺点：**
- 需要预知作业运行时间（实际很难）
- 对长作业不利，可能产生饥饿
- 不公平

**示例：**
```
作业序列：A(8ms) -> B(4ms) -> C(2ms)
执行顺序：C -> B -> A
周转时间：C=2, B=6, A=14
平均周转时间：(2+6+14)/3 = 7.33ms
```

---

### 3. 最短剩余时间优先（SRTF, Shortest Remaining Time First）

**算法原理：**
- SJF的抢占式版本
- 当新作业到达时，比较其运行时间与当前作业的剩余时间
- 如果新作业更短，则抢占CPU

**算法流程：**
1. 当前作业正在执行
2. 新作业到达，计算其运行时间
3. 比较新作业运行时间与当前作业剩余时间
4. 如果新作业更短，保存当前作业状态，切换到新作业
5. 否则，新作业加入就绪队列
6. 重复步骤2-5

**优点：**
- 比非抢占式SJF平均等待时间更短
- 对短作业响应快

**缺点：**
- 长作业可能长时间得不到执行
- 需要频繁进行剩余时间计算
- 上下文切换开销大

**示例：**
```
时刻0: A到达(8ms)，开始执行A
时刻2: B到达(4ms)，A剩余6ms > B的4ms，切换到B
时刻6: B完成，继续执行A
时刻12: A完成
```

---

### 4. 最高响应比优先（HRRN, Highest Response Ratio Next）

**算法原理：**
- 综合考虑等待时间和运行时间
- 响应比 = (等待时间 + 运行时间) / 运行时间 = 1 + 等待时间/运行时间
- 选择响应比最高的作业执行

**算法流程：**
1. 计算所有就绪作业的响应比
2. 选择响应比最高的作业执行
3. 作业完成后，重新计算剩余作业的响应比
4. 重复步骤2-3

**优点：**
- 兼顾短作业和长作业
- 短作业容易获得高响应比
- 长作业等待时间长后也能获得高响应比，避免饥饿

**缺点：**
- 需要预知运行时间
- 每次调度都要计算响应比，开销较大

**示例：**
```
作业A: 等待10ms, 运行5ms, 响应比 = 1 + 10/5 = 3
作业B: 等待5ms, 运行2ms, 响应比 = 1 + 5/2 = 3.5
作业C: 等待8ms, 运行8ms, 响应比 = 1 + 8/8 = 2
选择B执行（响应比最高）
```

---

### 5. 优先级调度算法

**算法原理：**
- 每个作业/进程分配一个优先级
- 优先级高的先执行
- 可以是静态优先级或动态优先级
- 可以是抢占式或非抢占式

**算法流程：**
1. 为每个作业/进程分配优先级
2. 按优先级排序就绪队列
3. 选择优先级最高的执行
4. 动态优先级：根据运行情况调整优先级
5. 重复步骤2-4

**静态优先级：**
- 创建时确定，运行过程中不变
- 简单，开销小
- 可能导致低优先级进程饥饿

**动态优先级：**
- 根据进程行为动态调整
- 等待时间长的提升优先级
- I/O密集型进程提升优先级
- 避免饥饿

**优点：**
- 灵活，可根据需求设置优先级
- 重要任务优先处理

**缺点：**
- 低优先级可能饥饿
- 优先级设置困难

---

### 6. 时间片轮转调度算法（RR, Round Robin）

**算法原理：**
- 将CPU时间划分为时间片（如10ms）
- 就绪队列按FCFS排列
- 每个进程执行一个时间片
- 时间片用完后，进程移到队列尾部
- 适用于分时系统

**算法流程：**
1. 设置时间片大小q
2. 从就绪队列头部取出进程
3. 进程执行时间片q
4. 时间片结束：
   - 如果进程未完成，移到队列尾部
   - 如果进程完成，退出
5. 重复步骤2-4

**时间片选择：**
- 太大：退化为FCFS
- 太小：上下文切换开销大
- 一般选择10-100ms

**优点：**
- 公平，响应时间好
- 适合交互式系统
- 不会饥饿

**缺点：**
- 平均周转时间较长
- 上下文切换开销
- 对I/O密集型进程不利

**示例：**
```
时间片q=4ms
进程A(10ms), B(6ms), C(2ms)
执行序列：
0-4: A(剩6ms)
4-8: B(剩2ms)
8-10: C(完成)
10-14: A(剩2ms)
14-16: B(完成)
16-18: A(完成)
```

---

### 7. 多级反馈队列调度算法

**算法原理：**
- 设置多个就绪队列，每个队列有不同优先级
- 高优先级队列时间片小，低优先级队列时间片大
- 新进程进入最高优先级队列
- 时间片用完未完成，降到下一级队列
- 综合了多种调度算法的优点

**算法流程：**
1. 设置n个队列Q1, Q2, ..., Qn
2. Q1优先级最高，时间片最小
3. Qn优先级最低，时间片最大（或FCFS）
4. 新进程进入Q1
5. 调度规则：
   - 优先调度高优先级队列
   - 同级队列内使用时间片轮转
6. 进程在Qi用完时间片未完成，降到Qi+1
7. I/O完成的进程可提升优先级

**队列设置示例：**
```
Q1: 时间片 = 8ms
Q2: 时间片 = 16ms
Q3: FCFS
```

**优点：**
- 兼顾短进程和长进程
- I/O密集型进程优先级高
- 响应时间好
- 不会饥饿

**缺点：**
- 实现复杂
- 参数设置困难

**执行示例：**
```
进程A(30ms)到达：
0-8: 在Q1执行8ms，降到Q2
8-24: 在Q2执行16ms，降到Q3
24-30: 在Q3执行6ms完成
```

## 第三章 进程同步与互斥 - 同步算法

### 8. Peterson算法

**算法原理：**
- 纯软件实现的两进程互斥算法
- 使用两个标志位和一个轮转变量
- 解决临界区互斥访问问题

**数据结构：**
```c
boolean flag[2];  // 表示进程是否想进入临界区
int turn;         // 表示轮到哪个进程
```

**算法流程：**

**进程Pi进入临界区：**
1. flag[i] = true;  // 表示Pi想进入
2. turn = j;        // 谦让，让对方先进
3. while(flag[j] && turn == j);  // 等待
4. 进入临界区
5. 执行临界区代码
6. flag[i] = false;  // 退出临界区

**互斥性证明：**
- 假设P0和P1同时在临界区
- 则flag[0]=true, flag[1]=true
- 但turn只能是0或1
- 如果turn=0，则P1在while循环等待
- 矛盾，所以不可能同时进入

**优点：**
- 纯软件实现
- 保证互斥、有空让进、有限等待

**缺点：**
- 忙等待，浪费CPU
- 只适用于两个进程
- 不适用于多处理器

### 9. 银行家算法（Banker's Algorithm）

**算法原理：**
- 死锁避免算法
- 在资源分配前检查系统是否处于安全状态
- 只有安全状态才分配资源
- 由Dijkstra提出

**数据结构：**
```
Available[m]: 可用资源向量，m种资源
Max[n][m]: 最大需求矩阵，n个进程
Allocation[n][m]: 已分配矩阵
Need[n][m]: 还需要矩阵，Need = Max - Allocation
Request[n][m]: 请求矩阵
```

**算法流程：**

**资源请求算法：**
1. 进程Pi请求资源Request[i]
2. 检查Request[i] <= Need[i]，否则出错
3. 检查Request[i] <= Available，否则等待
4. 试探性分配：
   ```
   Available = Available - Request[i]
   Allocation[i] = Allocation[i] + Request[i]
   Need[i] = Need[i] - Request[i]
   ```
5. 执行安全性检查
6. 如果安全，正式分配；否则恢复，Pi等待

**安全性检查算法：**
1. 初始化：
   ```
   Work = Available
   Finish[i] = false (所有进程)
   ```
2. 查找满足条件的进程i：
   ```
   Finish[i] == false
   Need[i] <= Work
   ```
3. 如果找到：
   ```
   Work = Work + Allocation[i]
   Finish[i] = true
   转步骤2
   ```
4. 如果所有Finish[i] == true，系统安全
5. 否则，系统不安全

**示例：**
```
3个进程P0, P1, P2
3种资源A(10), B(5), C(7)

        Max      Allocation    Need      Available
      A  B  C    A  B  C      A  B  C    A  B  C
P0    7  5  3    0  1  0      7  4  3    3  3  2
P1    3  2  2    2  0  0      1  2  2
P2    9  0  2    3  0  2      6  0  0

安全性检查：
1. Work = (3,3,2)
2. 找P1: Need(1,2,2) <= Work(3,3,2)，执行P1
   Work = (3,3,2) + (2,0,0) = (5,3,2)
3. 找P0: Need(7,4,3) > Work(5,3,2)，跳过
   找P2: Need(6,0,0) <= Work(5,3,2)，执行P2
   Work = (5,3,2) + (3,0,2) = (8,3,4)
4. 找P0: Need(7,4,3) <= Work(8,3,4)，执行P0
   Work = (8,3,4) + (0,1,0) = (8,4,4)
5. 所有进程完成，安全序列：P1 -> P2 -> P0
```

**优点：**
- 可以避免死锁
- 资源利用率高

**缺点：**
- 需要预知最大需求
- 进程数和资源数固定
- 开销大

### 10. 死锁检测算法

**算法原理：**
- 允许死锁发生，定期检测
- 使用资源分配图检测环路
- 简化资源分配图判断死锁

**数据结构：**
```
Available[m]: 可用资源向量
Allocation[n][m]: 已分配矩阵
Request[n][m]: 请求矩阵
Finish[n]: 进程是否能完成
```

**算法流程：**

1. 初始化：
   ```
   Work = Available
   对于所有i:
     如果Allocation[i] != 0，Finish[i] = false
     否则Finish[i] = true
   ```

2. 查找满足条件的进程i：
   ```
   Finish[i] == false
   Request[i] <= Work
   ```

3. 如果找到：
   ```
   Work = Work + Allocation[i]
   Finish[i] = true
   转步骤2
   ```

4. 如果存在Finish[i] == false，则系统死锁
   这些进程构成死锁进程集

**资源分配图简化：**

1. 找一个既不阻塞又非孤立的进程结点
2. 消去该进程的请求边和分配边
3. 将该进程变为孤立结点
4. 重复步骤1-3
5. 如果能完全简化，无死锁
6. 否则，不可简化的进程构成死锁

**示例：**
```
进程-资源分配图：
P1 -> R1 -> P2 -> R2 -> P1  (环路，死锁)

简化过程：
1. P3不阻塞，消去P3
2. P1和P2形成环路，无法简化
3. 检测到死锁：{P1, P2}
```

**优点：**
- 不限制资源请求
- 可以检测出死锁

**缺点：**
- 检测开销大
- 需要死锁恢复机制

### 11. 信号量PV操作

**算法原理：**
- 信号量S是整型变量
- 只能通过P、V操作访问
- P操作：申请资源（Wait）
- V操作：释放资源（Signal）
- 原语操作，不可中断

**数据结构：**
```c
typedef struct {
    int value;        // 信号量值
    struct process *queue;  // 等待队列
} semaphore;
```

**P操作流程：**
```
P(S):
    S.value--;
    if (S.value < 0) {
        将进程加入S.queue
        阻塞该进程
    }
```

**V操作流程：**
```
V(S):
    S.value++;
    if (S.value <= 0) {
        从S.queue中取出一个进程
        唤醒该进程
    }
```

**信号量含义：**
- S > 0: 可用资源数
- S = 0: 无资源，无等待
- S < 0: |S|为等待进程数

**互斥实现：**
```c
semaphore mutex = 1;

进程P:
    P(mutex);
    临界区;
    V(mutex);
```

**同步实现：**
```c
semaphore S = 0;

进程P1:
    操作A;
    V(S);

进程P2:
    P(S);
    操作B;  // B必须在A之后
```

**生产者-消费者问题：**
```c
semaphore empty = n;  // 空缓冲区
semaphore full = 0;   // 满缓冲区
semaphore mutex = 1;  // 互斥

生产者:
    P(empty);
    P(mutex);
    生产产品到缓冲区;
    V(mutex);
    V(full);

消费者:
    P(full);
    P(mutex);
    从缓冲区取产品;
    V(mutex);
    V(empty);
```

**注意事项：**
- P、V必须成对出现
- P操作顺序不能颠倒（先资源后互斥）
- 避免死锁

## 第四章 存储管理 - 分配算法

### 12. 最先适应算法（First Fit）

**算法原理：**
- 从空闲分区链表头开始查找
- 找到第一个满足要求的分区就分配
- 简单快速

**算法流程：**
1. 从空闲分区表头开始
2. 顺序查找第一个大小 >= 请求大小的分区
3. 如果找到：
   - 如果分区大小 == 请求大小，整个分配
   - 否则，分割分区，分配所需部分
4. 如果未找到，分配失败

**数据结构：**
```c
struct FreeBlock {
    int start;      // 起始地址
    int size;       // 大小
    struct FreeBlock *next;
};
```

**示例：**
```
空闲分区：20K, 10K, 30K, 15K
请求12K：
查找顺序：20K(满足) -> 分配12K，剩余8K
结果：8K, 10K, 30K, 15K
```

**优点：**
- 算法简单，速度快
- 倾向于保留大的空闲分区在高地址

**缺点：**
- 低地址产生很多小碎片
- 每次都从头查找，效率低

---

### 13. 下次适应算法（Next Fit）

**算法原理：**
- First Fit的改进
- 从上次分配位置开始查找
- 循环查找

**算法流程：**
1. 从上次分配结束位置开始
2. 顺序查找第一个满足要求的分区
3. 如果到链表尾，从头继续
4. 找到后分配，记录分配位置

**示例：**
```
空闲分区：20K, 10K, 30K, 15K
上次分配在10K后
请求12K：
从30K开始查找 -> 30K满足 -> 分配12K
下次从30K剩余部分开始
```

**优点：**
- 分配均匀，不集中在低地址
- 查找效率比First Fit高

**缺点：**
- 大的空闲分区容易被分割
- 缺乏大的空闲分区

---

### 14. 最优适应算法（Best Fit）

**算法原理：**
- 查找大小最接近请求的空闲分区
- 使剩余碎片最小
- 空闲分区按大小递增排序

**算法流程：**
1. 遍历所有空闲分区
2. 找到大小 >= 请求且最小的分区
3. 分配该分区

**示例：**
```
空闲分区（已排序）：10K, 15K, 20K, 30K
请求12K：
查找：10K(不够), 15K(满足且最小)
分配15K，剩余3K
```

**优点：**
- 避免大材小用
- 保留大的空闲分区

**缺点：**
- 产生很多小碎片
- 需要排序或全部查找，速度慢

---

### 15. 最坏适应算法（Worst Fit）

**算法原理：**
- 选择最大的空闲分区分配
- 使剩余分区尽可能大
- 空闲分区按大小递减排序

**算法流程：**
1. 查找最大的空闲分区
2. 如果满足要求，分配
3. 剩余部分仍然较大，可继续使用

**示例：**
```
空闲分区（递减）：30K, 20K, 15K, 10K
请求12K：
选择30K（最大）
分配12K，剩余18K
```

**优点：**
- 剩余分区较大，可用性好
- 减少小碎片

**缺点：**
- 大分区很快被用完
- 不利于大作业

---

### 16. 快速适应算法

**算法原理：**
- 将空闲分区按大小分类
- 为常用大小设置专门链表
- 快速找到合适分区

**数据结构：**
```
链表1: 4K分区
链表2: 8K分区
链表3: 16K分区
...
链表n: 其他大小
```

**算法流程：**
1. 根据请求大小，选择对应链表
2. 从链表中取出一个分区
3. 如果该链表为空，查找更大的链表
4. 分配分区

**优点：**
- 查找速度快
- 适合固定大小请求

**缺点：**
- 内存碎片回收困难
- 合并空闲分区复杂

---

### 17. 内存紧凑算法（Memory Compaction）

**算法原理：**
- 移动内存中的程序
- 将分散的空闲分区合并
- 消除外部碎片

**算法流程：**
1. 扫描内存，找出所有已分配分区
2. 将所有已分配分区移到内存一端
3. 合并所有空闲分区到另一端
4. 更新进程的地址映射

**移动策略：**

**策略1：向低地址移动**
```
移动前：[P1][空][P2][空][P3][空]
移动后：[P1][P2][P3][空闲区域]
```

**策略2：向高地址移动**
```
移动前：[空][P1][空][P2][空][P3]
移动后：[空闲区域][P1][P2][P3]
```

**策略3：向中间移动**
```
移动前：[P1][空][P2][空][P3][空]
移动后：[P1][P2][空闲区域][P3]
```

**实施时机：**
1. 某分区释放后立即紧缩
2. 找不到足够大的空闲分区时紧缩
3. 定期紧缩

**优点：**
- 消除外部碎片
- 提高内存利用率

**缺点：**
- 移动开销大
- 需要动态重定位支持
- 系统开销大

**示例：**
```
初始状态：
地址  内容
0     P1(10K)
10    空闲(5K)
15    P2(8K)
23    空闲(7K)
30    P3(12K)

紧凑后：
0     P1(10K)
10    P2(8K)
18    P3(12K)
30    空闲(12K)
```

### 18. 最优页面置换算法（OPT, Optimal）

**算法原理：**
- 理论最优算法，实际无法实现
- 置换将来最长时间不使用的页面
- 用于评估其他算法的性能

**算法流程：**
1. 发生缺页中断
2. 查看内存中所有页面
3. 预测每个页面下次访问时间
4. 选择最晚访问的页面置换
5. 调入新页面

**示例：**
```
页面访问序列：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2
物理块数：3

时刻  访问  内存状态        缺页
1     7     [7, -, -]       ✓
2     0     [7, 0, -]       ✓
3     1     [7, 0, 1]       ✓
4     2     [2, 0, 1]       ✓ (置换7，因为7最晚使用)
5     0     [2, 0, 1]       
6     3     [2, 0, 3]       ✓ (置换1)
7     0     [2, 0, 3]       
8     4     [4, 0, 3]       ✓ (置换2)
9     2     [4, 0, 2]       ✓ (置换3)
10    3     [4, 3, 2]       ✓ (置换0)
11    0     [0, 3, 2]       ✓ (置换4)
12    3     [0, 3, 2]       
13    2     [0, 3, 2]       

缺页次数：9次
```

**优点：**
- 缺页率最低（理论最优）

**缺点：**
- 无法预知未来访问序列
- 实际无法实现
- 仅用于性能比较

---

### 19. 先进先出页面置换算法（FIFO）

**算法原理：**
- 最简单的页面置换算法
- 置换最早进入内存的页面
- 使用队列实现

**算法流程：**
1. 维护一个页面队列，按进入时间排序
2. 发生缺页中断
3. 如果内存未满，新页面加入队尾
4. 如果内存已满，置换队首页面
5. 新页面加入队尾

**数据结构：**
```c
队列：[页面1] -> [页面2] -> [页面3] -> ...
      队首                        队尾
```

**示例：**
```
页面访问序列：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2
物理块数：3

时刻  访问  内存状态        缺页  队列
1     7     [7, -, -]       ✓     7
2     0     [7, 0, -]       ✓     7,0
3     1     [7, 0, 1]       ✓     7,0,1
4     2     [2, 0, 1]       ✓     0,1,2 (置换7)
5     0     [2, 0, 1]             
6     3     [2, 3, 1]       ✓     1,2,3 (置换0)
7     0     [2, 3, 0]       ✓     2,3,0 (置换1)
8     4     [4, 3, 0]       ✓     3,0,4 (置换2)
9     2     [4, 2, 0]       ✓     0,4,2 (置换3)
10    3     [4, 2, 3]       ✓     4,2,3 (置换0)
11    0     [0, 2, 3]       ✓     2,3,0 (置换4)
12    3     [0, 2, 3]             
13    2     [0, 2, 3]             

缺页次数：10次
```

**Belady异常：**
- 物理块增加，缺页率反而上升
- FIFO算法存在此异常

**优点：**
- 实现简单
- 开销小

**缺点：**
- 性能较差
- 可能置换经常使用的页面
- 存在Belady异常

### 20. 最近最少使用页面置换算法（LRU, Least Recently Used）

**算法原理：**
- 置换最长时间未被访问的页面
- 基于局部性原理：最近使用的页面将来可能还会使用
- 性能接近OPT

**算法流程：**
1. 为每个页面记录最后访问时间
2. 发生缺页中断
3. 查找最长时间未访问的页面
4. 置换该页面
5. 更新新页面的访问时间

**实现方法：**

**方法1：计数器**
```c
struct Page {
    int pageNum;
    int counter;  // 访问时间戳
};
```

**方法2：栈**
```
访问页面时，将其移到栈顶
栈底的页面是最久未使用的
```

**方法3：链表**
```
访问页面时，将其移到链表头
链表尾的页面是最久未使用的
```

**示例：**
```
页面访问序列：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2
物理块数：3

时刻  访问  内存状态        缺页  最近使用顺序
1     7     [7, -, -]       ✓     7
2     0     [7, 0, -]       ✓     0,7
3     1     [7, 0, 1]       ✓     1,0,7
4     2     [2, 0, 1]       ✓     2,1,0 (置换7)
5     0     [2, 0, 1]             0,2,1
6     3     [2, 0, 3]       ✓     3,0,2 (置换1)
7     0     [2, 0, 3]             0,3,2
8     4     [4, 0, 3]       ✓     4,0,3 (置换2)
9     2     [4, 0, 2]       ✓     2,0,4 (置换3)
10    3     [3, 0, 2]       ✓     3,2,0 (置换4)
11    0     [3, 0, 2]             0,3,2
12    3     [3, 0, 2]             3,0,2
13    2     [3, 0, 2]             2,3,0

缺页次数：8次
```

**优点：**
- 性能好，接近OPT
- 考虑了页面使用历史
- 不存在Belady异常

**缺点：**
- 实现复杂
- 开销大（需要记录访问时间）
- 硬件支持要求高

---

### 21. 时钟页面置换算法（Clock / 二次机会算法）

**算法原理：**
- LRU的近似算法
- 使用访问位（引用位）
- 循环扫描，给页面第二次机会

**算法流程：**
1. 页面组织成环形链表
2. 使用指针指向最老的页面
3. 发生缺页中断：
   - 检查指针指向页面的访问位
   - 如果访问位=1，置为0，指针前移（给第二次机会）
   - 如果访问位=0，置换该页面
4. 页面被访问时，访问位置1

**数据结构：**
```
环形链表：
[页面0, 访问位] -> [页面1, 访问位] -> [页面2, 访问位] -> ...
     ↑                                                    |
     |____________________________________________________|
     指针
```

**示例：**
```
物理块数：3
初始：[A,0] -> [B,0] -> [C,0]
      ↑指针

访问D：
1. 检查A，访问位=0，置换A
2. 结果：[D,1] -> [B,0] -> [C,0]
         指针移到B

访问B：
1. B在内存，访问位置1
2. 结果：[D,1] -> [B,1] -> [C,0]

访问E：
1. 检查B，访问位=1，置0，指针前移
2. 检查C，访问位=0，置换C
3. 结果：[D,1] -> [B,0] -> [E,1]
                           ↑指针
```

**改进：增强型时钟算法**
使用访问位和修改位：
- (0,0)：最佳置换候选
- (0,1)：次佳，但需要写回
- (1,0)：最近访问过
- (1,1)：最近访问且修改

**优点：**
- 实现简单
- 开销小
- 性能接近LRU

**缺点：**
- 性能略低于LRU
- 最坏情况需要扫描两遍

---

### 22. 最不常用页面置换算法（LFU, Least Frequently Used）

**算法原理：**
- 置换访问次数最少的页面
- 为每个页面维护访问计数器
- 认为访问次数少的页面将来也不常用

**算法流程：**
1. 为每个页面维护访问计数器
2. 页面被访问时，计数器加1
3. 发生缺页中断
4. 选择计数器值最小的页面置换
5. 新页面计数器初始化为1

**数据结构：**
```c
struct Page {
    int pageNum;
    int counter;  // 访问次数
};
```

**示例：**
```
页面访问序列：7, 0, 1, 2, 0, 3, 0
物理块数：3

时刻  访问  内存状态              缺页  计数器
1     7     [7(1), -, -]          ✓     7:1
2     0     [7(1), 0(1), -]       ✓     7:1, 0:1
3     1     [7(1), 0(1), 1(1)]    ✓     7:1, 0:1, 1:1
4     2     [2(1), 0(1), 1(1)]    ✓     2:1, 0:1, 1:1 (置换7)
5     0     [2(1), 0(2), 1(1)]          2:1, 0:2, 1:1
6     3     [2(1), 0(2), 3(1)]    ✓     2:1, 0:2, 3:1 (置换1)
7     0     [2(1), 0(3), 3(1)]          2:1, 0:3, 3:1
```

**优点：**
- 考虑了页面使用频率
- 对某些访问模式效果好

**缺点：**
- 早期频繁访问的页面难以置换
- 不能反映最近的访问模式
- 开销大

---

## 页面置换算法性能比较

**缺页率比较（一般情况）：**
```
OPT < LRU ≈ Clock < FIFO < LFU（取决于访问模式）
```

**实现复杂度：**
```
FIFO < Clock < LFU < LRU < OPT（无法实现）
```

**实际应用：**
- **FIFO**：简单系统
- **Clock**：大多数操作系统（如Linux）
- **LRU**：高性能系统（需硬件支持）
- **改进Clock**：Windows等现代操作系统

**选择建议：**
1. 性能要求高：LRU或改进Clock
2. 实现简单：FIFO或Clock
3. 硬件支持有限：Clock
4. 理论分析：OPT作为基准

## 第五章 设备管理 - 磁盘调度算法

磁盘调度算法用于优化磁盘访问性能，减少磁头移动时间。

### 磁盘访问时间组成
```
总访问时间 = 寻道时间 + 旋转延迟 + 传输时间
- 寻道时间：磁头移动到目标磁道的时间（主要优化目标）
- 旋转延迟：磁盘旋转到目标扇区的时间
- 传输时间：数据传输的时间
```

---

### 23. 先来先服务磁盘调度算法（FCFS）

**算法原理：**
- 按照磁盘I/O请求到达的顺序进行服务
- 最简单、最公平的调度算法
- 不考虑磁头当前位置

**算法流程：**
1. 维护一个磁盘请求队列
2. 按照请求到达顺序排队
3. 依次处理每个请求
4. 磁头按请求顺序移动

**示例：**
```
磁盘请求队列：98, 183, 37, 122, 14, 124, 65, 67
当前磁头位置：53
磁道范围：0-199

访问顺序：53 -> 98 -> 183 -> 37 -> 122 -> 14 -> 124 -> 65 -> 67

磁头移动距离：
53->98:   45
98->183:  85
183->37:  146
37->122:  85
122->14:  108
14->124:  110
124->65:  59
65->67:   2
总移动距离：640个磁道
平均移动距离：640/8 = 80个磁道
```

**移动轨迹图：**
```
磁道号

    |   /|
    |  / |

65  |    |     /
53  |*   |    /
37  |    |*  /
14  |    | */
0   |____|_____________
    时间顺序
```

**优点：**
- 实现简单
- 公平，不会产生饥饿
- 适合请求较少的系统

**缺点：**
- 磁头移动距离大
- 平均寻道时间长
- 效率低

### 24. 最短寻道时间优先算法（SSTF, Shortest Seek Time First）

**算法原理：**
- 选择距离当前磁头位置最近的请求
- 优先处理寻道时间最短的请求
- 贪心算法思想

**算法流程：**
1. 计算所有请求与当前磁头位置的距离
2. 选择距离最短的请求
3. 移动磁头到该位置，处理请求
4. 更新当前位置
5. 重复步骤1-4

   -> 98(距离84) -> 122(距离24) -> 124(距离2) -> 183(距离59)

磁头移动距离：
53->65:   12
65->67:   2
67->37:   30
37->14:   23
14->98:   84
98->122:  24
122->124: 2
124->183: 59
总移动距离：236个磁道
平均移动距离：236/8 = 29.5个磁道
```

**优点：**

**缺点：**

- 不公平
- 对中间磁道的请求有利

### 25. 扫描算法（SCAN / 电梯算法）

**算法原理：**
- 磁头在磁盘上来回扫描
- 移动方向上处理所有请求
- 到达边界后改变方向
- 类似电梯运行方式

**算法流程：**
1. 确定磁头移动方向（向内或向外）
2. 沿当前方向移动
3. 处理路径上的所有请求
4. 到达边界（0或199）后反向
5. 继续处理反向路径上的请求

**示例：**

访问顺序：
53 -> 37(向内) -> 14(向内) -> 0(到达边界) 
   -> 65(向外) -> 67(向外) -> 98(向外) -> 122(向外) -> 124(向外) -> 183(向外)

磁头移动距离：
53->37:   16
37->14:   23
14->0:    14
0->65:    65
65->67:   2
67->98:   31
98->122:  24
122->124: 2
124->183: 59
总移动距离：236个磁道
```

**移动轨迹图：**
```
磁道号
199 |
183 |              *
    |             /
124 |           */
122 |          *
98  |        */
67  |      */
65  |     *
53  |*   /
37  | * /
14  |  */
0   |   *___________
    时间顺序
    ↓向内  ↑向外
```

**优点：**
- 避免饥饿
- 响应时间方差小
- 吞吐量高

**缺点：**
- 对两端磁道不公平（等待时间长）
- 平均寻道时间略高于SSTF

### 26. 循环扫描算法（C-SCAN, Circular SCAN）

**算法原理：**
- SCAN的改进版本
- 磁头只在一个方向上服务请求
- 到达边界后，快速返回另一端（不服务）
- 将磁盘看作环形结构

**算法流程：**
1. 磁头沿一个方向移动（如向外）
2. 处理路径上的所有请求
3. 到达最外端后，快速返回最内端（不处理请求）
4. 继续向外移动，处理请求
5. 重复步骤2-4

**示例：**
```
磁盘请求队列：98, 183, 37, 122, 14, 124, 65, 67
当前磁头位置：53
移动方向：向磁道199移动
磁道范围：0-199

访问顺序：
53 -> 65(向外) -> 67(向外) -> 98(向外) -> 122(向外) -> 124(向外) -> 183(向外)
   -> 199(到达边界) -> 0(快速返回) -> 14(向外) -> 37(向外)

磁头移动距离：
53->65:   12
65->67:   2
67->98:   31
98->122:  24
122->124: 2
124->183: 59
183->199: 16
199->0:   199 (快速返回，不服务)
0->14:    14
14->37:   23
总移动距离：382个磁道
```

**移动轨迹图：**
```
磁道号
199 |      *
183 |     */
    |    /
124 |  */
122 | *
98  |*/
67  |*
65  |*
53  |*
37  |        *
14  |       */
0   |      *_________
    时间顺序
    ↑向外  ↓快速返回 ↑向外
```

**优点：**
- 对所有磁道更公平
- 响应时间方差更小
- 避免了SCAN对两端的不公平

**缺点：**
- 返回时间是额外开销
- 总移动距离可能比SCAN长

---

### 27. LOOK调度算法

**算法原理：**
- SCAN的改进版本
- 磁头移动到最远请求位置即反向
- 不必到达磁盘边界
- "边走边看"（LOOK）

**算法流程：**
1. 确定移动方向
2. 沿当前方向移动
3. 处理路径上的所有请求
4. 到达该方向最远请求后立即反向
5. 不必到达磁盘边界

**示例：**
```
磁盘请求队列：98, 183, 37, 122, 14, 124, 65, 67
当前磁头位置：53
初始方向：向磁道0移动
磁道范围：0-199

访问顺序：
53 -> 37(向内) -> 14(向内，最远请求) 
   -> 65(向外) -> 67(向外) -> 98(向外) -> 122(向外) -> 124(向外) -> 183(向外，最远请求)

磁头移动距离：
53->37:   16
37->14:   23
14->65:   51
65->67:   2
67->98:   31
98->122:  24
122->124: 2
124->183: 59
总移动距离：208个磁道
```

**移动轨迹图：**
```
磁道号
199 |
183 |              * (最远请求)
    |             /
124 |           */
122 |          *
98  |        */
67  |      */
65  |     *
53  |*   /
37  | * /
14  |  *  (最远请求)
0   |________________
    时间顺序
    ↓向内  ↑向外
```

**优点：**
- 比SCAN减少了不必要的移动
- 平均寻道时间更短
- 避免饥饿

**缺点：**
- 实现稍复杂（需要知道最远请求）

### 28. C-LOOK调度算法（循环LOOK）

**算法原理：**
- C-SCAN和LOOK的结合
- 单向服务，到最远请求后返回
- 返回到另一方向的最远请求位置
- 不必到达磁盘边界

**算法流程：**
1. 磁头沿一个方向移动
2. 处理路径上的所有请求
3. 到达该方向最远请求后
4. 快速返回到反方向的最远请求位置
5. 继续单向服务

**示例：**
```
磁盘请求队列：98, 183, 37, 122, 14, 124, 65, 67
当前磁头位置：53
移动方向：向磁道199移动
磁道范围：0-199

访问顺序：
53 -> 65(向外) -> 67(向外) -> 98(向外) -> 122(向外) -> 124(向外) -> 183(向外，最远)
   -> 14(快速返回到最远内侧请求) -> 37(向外)

磁头移动距离：
53->65:   12
65->67:   2
67->98:   31
98->122:  24
122->124: 2
124->183: 59
183->14:  169 (快速返回)
14->37:   23
总移动距离：322个磁道
```

**移动轨迹图：**
```
磁道号
199 |
183 |     * (最远请求)
    |    /|
124 |  */ |
122 | *   |
98  |*/   |
67  |*    |
65  |*    |
53  |*    |
37  |      *
14  |     */
0   |________________
    时间顺序
    ↑向外  ↓快速返回 ↑向外
```

**优点：**
- 比C-SCAN减少不必要移动
- 对所有磁道公平
- 平均寻道时间短

**缺点：**
- 实现最复杂
- 需要维护请求的最远位置

### 选择建议

- 使用C-SCAN或C-LOOK
